OpenAI buys Ona to turn Codex from a per-session helper into a persistent production agent

OpenAI agreed on Thursday to acquire Ona, the secure cloud-sandbox and orchestration startup whose two million developer accounts already deploy ephemeral, hypervisor-isolated environments for AI agents. Codex teams now run more than five million weekly users — up roughly fourfold over the first half of this year — and the deal hands them Ona's persistent-environment runtime, VPC deployment, audit logs and policy-enforcement plane in a single drop-in. Each Codex task already runs in its own isolated cloud sandbox; the Ona stack extends that primitive into long-running, multi-session agents that hold state across days rather than minutes. Founder and chief executive Johannes Landgraf joins OpenAI to lead the integration into the Codex division. Financial terms were not disclosed and the deal is subject to customary closing conditions.

OpenAI · 2026-06-11 · Source ↗ ·

OpenAI announced on June 11 that it had agreed to acquire Ona, a German-founded cloud platform that gives autonomous agents pre-configured, secure execution environments. The transaction folds Ona's engineering team and its execution runtime into the Codex division, the unit that ships OpenAI's coding agent and the standalone Codex CLI. Financial terms were withheld; the deal is subject to customary regulatory and closing conditions.

The strategic logic is straightforward. Codex today runs as a per-session sandbox: each task spins up its own isolated cloud environment, executes against the code, and shuts down when the session ends. That model works for the autocomplete-style assists Codex was built around, but it falls over the moment an organization tries to use Codex for the kind of work agentic deployment actually needs — multi-day refactors, scheduled maintenance jobs, persistent CI agents that watch a repo for weeks. Ona's product is exactly the missing primitive: hypervisor-isolated cloud environments with persistent filesystems, per-agent identities, OpenTelemetry tracing built in, and the policy-and-audit machinery enterprises insist on before they let an AI system anywhere near production.

"By bringing Ona to OpenAI, we will expand Codex beyond work tied to a single device or active session and help more organizations deploy agents securely in production."— OpenAI, official announcement

What Ona built

Ona was founded by Johannes Landgraf and the team behind Gitpod, the cloud development environment that Ona pivoted from in late 2025 once it became clear the more lucrative customer was the autonomous agent rather than the human developer it was originally built for. Ona today claims roughly two million developer accounts using its sandbox runtime, with the platform's central abstraction being a reproducible cloud environment that an AI agent can boot, mutate, snapshot and tear down through an API. Customers run Ona inside their own virtual private cloud with enterprise controls — RBAC, secret vaults, audit logs, and policy enforcement that can block a Codex task from touching specific resources or making egress requests outside an allowed list.

The five-million-weekly-Codex-user statistic OpenAI disclosed in the same announcement is the other half of the rationale. Codex usage has grown about fourfold since early 2026, with the company calling out research, analysis, build and automation as the dominant workloads. Each of those workloads benefits from the Ona pattern: a sandbox that boots in seconds, holds state across the lifetime of an investigation, and produces a forensic trail an auditor can read.

What changes for OpenAI customers

OpenAI framed the integration as bringing Ona's "secure execution and orchestration technology" inside Codex, full stop. In practice that should let a Codex task survive across sessions, share state with downstream tasks, and run inside an enterprise's own VPC. The audit-log and policy-enforcement layers map directly onto the kinds of compliance controls regulated industries have been waiting for before they will green-light an AI agent on a production codebase.

"I always thought selling the company would feel like an ending. Instead, it feels like our life's work just got bigger and more important. Agents need more than intelligence; they need a trusted workspace."— Johannes Landgraf, Ona founder and chief executive

Landgraf's framing — agents need a trusted workspace, not just a smart model — is also the explicit pitch Microsoft made for Foundry Agent Service at Build 2026 two weeks ago, when it rolled hosted agents to general availability with hypervisor-isolated sandboxes, per-agent Entra IDs, OpenTelemetry tracing and sub-100ms cold starts. The convergence is not coincidence. The frontier labs and the hyperscalers have decided in parallel that the next year of agent revenue depends less on a smarter model than on production-grade runtime substrate underneath. OpenAI buying Ona is the cleanest single signal of that bet so far this quarter.

The Codex business behind the deal

Codex passed five million weekly users earlier this month, up about 400 percent from the start of the year by OpenAI's count, with enterprise rollouts at Apollo, Cisco, Goldman Sachs, Lowe's, Optum, OpenAI itself and others in regulated industries cited by Ona in adjacent customer disclosures. The Oracle Cloud Universal Credits pact that landed Wednesday — OCI customers can now spend their existing Oracle commitment on OpenAI frontier models and Codex through the OCI Marketplace — sits alongside this deal as the enterprise distribution channel. Combine the two and OpenAI now has both a credible enterprise procurement path and a runtime fit to host agents inside customer-controlled infrastructure.

For Ona's existing customers, the public language so far is continuity: the platform will continue serving its developer base inside OpenAI, with the team focused on integrating Ona's primitives into Codex. The acquisition closes pending regulatory review with no announced timeline.

Big picture

OpenAI buying Ona on the same week Codex passed five million weekly users and Oracle opened its Universal Credits to the model is a clean statement of where the agent market is moving: the bottleneck is no longer raw intelligence but the production runtime around it. Anthropic and Tata Consultancy Services hit the same theme from the partner side, signing a Global Premier deal that licenses Claude to 50,000 TCS associates and stands up a dedicated business unit for regulated industries in financial services, healthcare, telecom and aviation. Two miles down the road the financing layer is rearranging itself to match. SpaceX priced its IPO Thursday at a fixed $135 a share for a $75 billion raise at a $1.77 trillion valuation, the largest listing in history, ahead of Friday's Nasdaq debut under SPCX.

Around the edges the same agent-runtime thesis keeps showing up in different costumes. Cyera, the data-security operator whose Trust Layer governs what AI agents may see and do inside enterprise data estates, raised $600 million at a $12 billion valuation, quadrupling its price in 18 months. Cursor pushed Bugbot onto Composer 2.5 and made the average code-review 90 seconds instead of five minutes. xAI opened the Grok Build plugin marketplace to every SuperGrok and X Premium Plus subscriber. Anthropic and Microsoft co-disclosed a critical CVE in the Claude Code GitHub Action that exfiltrated CI secrets through unsanitized environment variables, then patched it. The substrate is hardening across vendors at the same time, because the same customers are asking for the same things.

Cyera quadruples its valuation to $12 billion on a $600 million Series F to build the AI trust layer

Cyera · 2026-06-11 · Source ↗

Lightspeed leads the round with Saudi Aramco's Prosperity7 strategic. CEO Yotam Segev says the platform now sits inside about 1,400 enterprises gating what AI agents may see and do across structured and unstructured data estates.

Cyera, the New York and Tel Aviv-based data security operator, disclosed Thursday a $600 million Series F at a $12 billion post-money valuation. Lightspeed Venture Partners led the round with Saudi Aramco's Prosperity7 Ventures joining as a strategic alongside Evolution Equity, B Capital, Gaingels and existing backers Greenoaks, Coatue and Sequoia. The valuation marks a quadrupling in 18 months, from $3 billion in November 2024 through $7 billion in June 2025 to $12 billion today.

Co-founder and chief executive Yotam Segev pitched the round on the back of the agent thesis the rest of the week's news keeps echoing. Cyera's product — what the company calls a Trust Layer — sits between AI agents and the data they reach across the enterprise. It maps where sensitive data lives, who and what can access it, and inserts an authoritative middle tier that an agent must consult before reading, writing or moving structured rows, unstructured documents, cloud storage objects or SaaS records. The "Trust Layer" framing replaces the more familiar Data Security Posture Management (DSPM) language Cyera came up under, signaling the company sees the AI agent control problem as a strictly larger market than the DSPM category it currently anchors.

What the platform actually does

Cyera ingests metadata and content samples from cloud data warehouses, SaaS systems, object stores and on-premises databases. It classifies what it finds against a continuously trained taxonomy of sensitive data types — PHI, PII, PCI, source code, customer correspondence, internal financials. It builds an access graph showing which identities and which agents currently have read or write paths to which classifications. Then it enforces policy at runtime, either through native cloud-provider hooks (BigQuery row-level filters, Snowflake row-access policies) or through an MCP-style mediation layer that intercepts agent tool calls before they fire.

The company says about 1,400 enterprises now run Cyera in production, with the Fortune 500 cohort the bulk of the late-stage revenue and the customer profile sliding rapidly toward "any company giving an AI agent access to anything sensitive." The Saudi Aramco strategic ties the round to the energy-major use case Cyera has been quietly building for two years, where regulatory exposure on production data is structurally higher than the average enterprise.

"AI is rewriting how every enterprise operates. The companies that win this next era will be the ones that can move fast on AI with confidence — and that requires a trust layer that knows where the data lives, who can use it, and how AI is allowed to act on it."— Yotam Segev, Cyera co-founder and chief executive

Why this round, why now

Three forces are pushing the valuation. First: the customer base is repricing data security to "agent governance" rates rather than legacy DSPM rates because the loss function is now agent-driven data exfiltration rather than human-driven mistakes. Second: the EU AI Act transparency rules fully enforceable from August 2 give regulated enterprises an external clock to bring an agent-governance system live. Third: the public-market comparable set, with Wiz acquired by Google for $32 billion and SentinelOne and CrowdStrike both trading at expanded multiples, lets growth investors underwrite a $12 billion private round on the assumption a public path arrives at $20 billion-plus on the back of the agent inflection.

The round is also notable for the cadence. Cyera raised $300 million in February at a $9 billion implied valuation, with the Series F coming four months later at a 33 percent step-up. That cadence is closer to a frontier-model lab than a security operator — closer to Anthropic's pace, in fact — which is itself a signal of how the market is pricing the agent-governance segment.

Non-GAAP earnings per share came in at $2.11, up 24 percent. Cloud revenue was the headline line: $9.9 billion in the quarter, up 47 percent in USD. Inside that figure, cloud infrastructure (IaaS) revenue grew 93 percent to $5.8 billion, and cloud applications (SaaS) grew 10 percent to $4.1 billion.

The full-year picture is the same story at scale. FY2026 revenue of $67.4 billion was up 17 percent. Full-year cloud revenue of $34.0 billion grew 39 percent, with cloud infrastructure of $18.1 billion up 77 percent. Non-GAAP full-year EPS came in at $7.63. Oracle raised the quarterly dividend by 25 percent to $0.50 a share, with a record date of July 10 and payment on July 24.

The number that matters

Remaining performance obligations — Oracle's backlog of contracted but un-recognized revenue — climbed to $638 billion at quarter end, up 363 percent year over year and a jaw-dropping $85 billion sequentially. Oracle co-chief executive Clayton Magouyrk attributed roughly $67 billion of the quarterly RPO jump to AI infrastructure contracts. Of that, about $75 billion of the total RPO is now tied to what Oracle described as "bring-your-own-hardware" or prepaid GPU structures, where the customer either supplies their own accelerators or pre-pays for the capacity Oracle stands up.

The bring-your-own-hardware framing matters because it is the cleanest signal yet that the hyperscaler-AI customer relationship is becoming a true co-investment. The customer — most often an OpenAI, Anthropic, Meta or xAI affiliate — commits the capital for the GPUs and Oracle commits the datacenter shell, networking, power and operations. The economics swap upfront capex risk for a longer-duration operating revenue stream and a lower expected RPO conversion margin than a vanilla cloud deal would imply.

The capital plan

Funding the build-out is the second story. Oracle disclosed it will raise roughly $40 billion in calendar 2027 via debt and equity sales. About $20 billion of that comes through an at-the-market equity offering, the first time Oracle has used the structure at this scale. The remainder is term debt. Importantly, Oracle said it does not plan to issue additional debt in calendar 2026 itself, having already raised $43 billion in FY2026.

"Oracle is the world's fastest growing provider of cloud datacenters."— Oracle, FY2026 press release

The capex cadence Oracle disclosed for FY2026 was $55.7 billion. Guidance for FY2027 calls for total revenue growth of 27 to 29 percent — implying around $86 billion in revenue — with cloud revenue growth of 57 to 63 percent at constant currency. Oracle is targeting full-year FY2027 revenue of $90 billion. Non-GAAP EPS guidance was lifted to $8.05.

Market reaction

Shares fell more than 10 percent in early Wednesday trade despite the headline beat. The selling concentrated on three concerns. First, the magnitude of the RPO leap raised questions about quality — specifically the prepaid and bring-your-own-hardware contracts that book at lower gross margins than traditional cloud. Second, the size of the ATM equity offering and the implied dilution. Third, the FY2027 cloud revenue guide came in below the most aggressive sell-side estimates, which had been calling for triple-digit cloud growth on the back of the OpenAI Stargate commitment.

Net of the noise, the print remains the cleanest single corporate validation this week of the AI infrastructure build cycle. Oracle's RPO trajectory — $85 billion of incremental backlog in a single quarter, sustained 47 percent cloud growth, an explicit $90 billion revenue target — sits alongside the Vera Rubin partner list and the Nvidia datacenter capex forecasts as evidence the spend is going from planned to contracted.

For competitive context: Microsoft has the equivalent through Azure OpenAI Service, Google through Vertex AI Model Garden, and AWS through Bedrock. Oracle adding OpenAI to Universal Credits closes the last credible hyperscaler distribution gap. The unanswered question is how this fits with OpenAI's Anthropic-rivaling enterprise sales motion, which has historically been a direct field force selling against the hyperscaler intermediaries. Cursor's Bugbot moves to Composer 2.5 and now finishes 90 percent of code reviews under three minutes

Cursor · 2026-06-10 · Source ↗

Average review time drops from five minutes to 90 seconds, cost per run falls 22 percent, and Bugbot now catches 10 percent more bugs per review (0.62 vs 0.56). New /review-bugbot and /review-security shortcuts run pre-push, with smarter PR re-reviews that diff against the last analysed state.

Cursor pushed a major upgrade to Bugbot, its AI code-review agent, late on June 10. The update moves Bugbot onto Composer 2.5, the agent-tuned variant of Cursor's in-house model. The performance numbers are the headline: average review time drops from roughly five minutes to about 90 seconds, cost per run falls 22 percent, and Bugbot catches an average of 0.62 bugs per review against the previous 0.56 — a 10 percent improvement on bug detection while running three times faster and cheaper.

Cursor framed the win as Composer 2.5 training paying off. The model was trained specifically on long-horizon code-review trajectories rather than the more general code-generation distribution earlier Composer checkpoints used. The training-data choice shows up in the shape of the speed gain: 90 percent of Bugbot runs now complete inside three minutes, which is the threshold past which developers consistently switch tasks and lose the review's context. Cutting below that line is the throughput win.

What's new

Three new features ship alongside the model upgrade. First, /review prompts the developer to pick which agents to run before pushing, with /review-bugbot and /review-security as direct shortcuts that skip the picker. Second, Bugbot can now be configured to only review what is new since the last review, keeping feedback focused on the latest changes rather than re-flagging issues the developer has already triaged. Third, GitHub and GitLab synchronization: if a developer runs /review locally and then opens a PR with the same diff, Bugbot recognizes the diff, skips the duplicate review, and leaves a comment noting it has already analysed that change.

Cursor said the speed and cost wins come "in large part" from Composer 2.5's training, with the rest from infrastructure improvements to the agent runtime. The model variant powering Bugbot is the same checkpoint Cursor is rolling into other Composer-backed agents inside the IDE, with CLI support coming. Bugbot is available on cursor.com/agents and inside Cursor 3.7 or later.

"In practice, 90 percent of Bugbot runs now finish in under three minutes."— Cursor changelog, June 10

Competitive context

Bugbot's positioning is a direct shot across the bow of Anthropic's Claude Code Reviews and GitHub Copilot's autonomous review agent, both of which have been racing on the same metric: time-to-first-review on a freshly opened PR. Cursor's claim of 90 seconds median undercuts the public numbers from either competitor by a healthy margin, and the cost reduction reframes the unit economics for organizations that have been gating wider rollout on per-review economics. The Bugbot updates also slot into the broader Cursor strategic message — that the company's in-house model work, not a wrapper around frontier-lab APIs, is now what's driving the product's speed and cost curve.

Worth noting: Bugbot's bug-detection rate of 0.62 bugs per review is a per-PR average, not a recall figure against a fixed benchmark. Cursor has not published a comparison against Bugbot's hit rate on a public eval such as CodeContests or a fixed bug-detection benchmark, which makes the head-to-head against Claude Code Reviews still apples-to-oranges. The throughput win, on the other hand, is verifiable in any developer's day-to-day use, and that is the metric Cursor's customer base prices.

The Rescana ThreatsDay Bulletin published on June 11 cataloged a critical-severity vulnerability in Anthropic's Claude Code GitHub Action that let attackers steal CI/CD secrets through a path the Action's sandbox model was supposed to close. The flaw, present in every version of the Action before 2.1.128, allowed an attacker who could trigger the Action to read /proc/self/environ inside the Action's container and walk away with whatever the host workflow had injected as environment variables — typically AWS IAM keys, database connection strings, signing keys and OAuth refresh tokens.

Microsoft's security research team found the bug and disclosed it responsibly to Anthropic, which patched the action in version 2.1.128 on May 5 and pushed a forced-update notice to all users running pre-2.1.128 versions. The ThreatsDay write-up published Thursday is the first detailed public post-mortem of the issue, which Rescana ranks as a "high-severity patch" alongside the Miasma supply-chain worm toolkit leak in its weekly threat round-up.

How the bug worked

The Claude Code GitHub Action ships as a containerised runner that the user invokes from a workflow YAML to run a Claude-driven code-review or code-generation step. The Action's design intent is that workflow secrets passed through the GitHub Actions secret-injection mechanism should be visible only to the workflow steps that explicitly reference them, with the Action itself unable to read secrets that have not been handed to it.

The design failed for an unfortunate Linux-implementation reason. GitHub Actions exposes secrets to step containers through environment variables. The Linux kernel maintains a copy of every process's environment under /proc/{pid}/environ, and a process can read its own copy through /proc/self/environ regardless of which environment variables the calling code thinks it has access to. The Action's container did not sanitize /proc/self/environ before handing control to the Claude model. A prompt-injection payload — or a malicious tool registered through MCP — could ask Claude to read the file, parse the null-separated key=value pairs, and either return them in the model's output or write them to an attacker-controlled endpoint.

"AI-powered agents, such as the OpenClaw email agent and the Claude Code GitHub Action by Anthropic, are susceptible to advanced phishing and social engineering tactics, with attackers exploiting these agents by crafting plausible requests that induced the AI to forward sensitive credentials to external adversaries."— Rescana ThreatsDay Bulletin, June 11

What the fix does

Version 2.1.128 of the Action wipes /proc/self/environ before the Claude runtime starts, restricts the in-container filesystem mount such that proc is not directly readable in the same way, and adds a final allow-list check on outbound network calls during a session — so an attacker who manages to get Claude to emit secrets cannot tunnel them to an exfiltration endpoint without first registering that endpoint with the workflow's egress policy. Anthropic also rotated the signing keys for the Action and added a startup attestation that surfaces the running version in the workflow log.

The wider lesson the ThreatsDay write-up draws is that the agent runtime is the security perimeter. The model itself is not vulnerable in the cryptographic-flaw sense; the runtime around it leaked secrets because the sandbox was incomplete. The CVE bracketed with the Miasma supply-chain worm leak (where a malicious npm package containing autonomous code-modification agents propagated through dependency graphs) makes the broader point that AI agents now sit on the threat-model map adjacent to traditional supply-chain attacks. Organizations running Claude Code GitHub Action should confirm they are on 2.1.128 or later. Anthropic's runtime hardening continues into the wider Claude Code stack.

The practical impact is that an iOS or macOS developer can write a single line of code against Foundation Models and have the runtime route to whichever provider best fits the request — Apple's on-device model for low-latency text-completion, Private Cloud Compute for medium-complexity work the device cannot serve, Anthropic's Claude for heavy agentic work, or Google's Gemini for multimodal video reasoning.
