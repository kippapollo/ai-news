Apple's third-generation Foundation Models ship across five tiers, with Cloud Pro running on NVIDIA Blackwell inside Google Cloud

Apple Machine Learning Research · 2026-06-12 · Source ↗

The AFM 3 family covers two on-device models (AFM 3 Core, AFM 3 Core Advanced) plus three Private Cloud Compute models (AFM 3 Cloud, ADM 3 Cloud image, AFM 3 Cloud Pro). Cloud Pro matches Gemini Frontier quality and runs on Blackwell B200 GPUs operated by Google under NVIDIA Confidential Computing attestation, with rollout to full capacity by end of summer 2026.

Apple closed out WWDC 2026 with the formal launch of its third-generation Apple Foundation Models. The release page on Apple Machine Learning Research describes the family as five models spanning two on-device tiers and three Private Cloud Compute tiers. The on-device pair is AFM 3 Core (the default model that ships inside every iOS 27, iPadOS 27 and macOS Golden Gate device) and AFM 3 Core Advanced (a 20-billion-parameter on-device model gated to M-series Macs and high-end iPad Pros). The cloud pair adds AFM 3 Cloud, ADM 3 Cloud (an image model), and AFM 3 Cloud Pro — the flagship tier Apple positions as comparable in quality to Google's Gemini Frontier.

The architectural disclosure that landed across Friday's developer sessions matters more than the model count. AFM 3 Cloud Pro does not run inside Apple's own data centers. It runs on NVIDIA Blackwell B200 GPUs sitting inside Google Cloud, with NVIDIA Confidential Computing providing the hardware-attested execution environment that Apple's Private Cloud Compute privacy guarantees rely on. Apple's PCC stack has been extended so that a request submitted from an iPhone is attested end-to-end through Apple's signing infrastructure and the Blackwell GPU's TEE before any inference happens. Rollout to full operational capacity is scheduled by the end of summer 2026, with a gradual ramp through July and August.

"NVIDIA GPUs will support server-side inference for Apple Foundation Models, custom-built by Apple and Google, leveraging the technologies behind the Gemini family of models."— NVIDIA, blog post on Confidential Computing for Apple Private Cloud Compute

Why Gemini is in the picture

Apple framed AFM 3 Cloud Pro as a co-built model, with Google contributing Gemini technology under license and Apple contributing its distillation, fine-tuning and privacy-routing layer. The arrangement is structurally similar to the way Apple licenses Google Maps API but operates the rendering layer itself: Apple owns the experience, Google contributes the weights underneath. Cloud Pro is the model that powers the most complex Siri requests, multi-step on-device Writing Tools tasks, and the visual reasoning behind Apple Intelligence's image-summary feature. Apple's roughly $1 billion annual licensing payment to Google for the underlying tech was first disclosed at the WWDC opening keynote on June 8.

Multimodal image input lands across all tiers

The Foundation Models framework session on Thursday confirmed multimodal image input is now a first-class parameter across all five AFM 3 models. Developers can pass an image plus text prompt and get grounded reasoning back, with the on-device tier handling labelling, OCR-style transcription and chart reading, while harder cases route to Cloud or Cloud Pro automatically. The session demoed a real-estate app that turns a camera capture of a room into a Claude-generated renovation plan in roughly three seconds round-trip.

"AFM Cloud Pro is the product of our collaboration with Google. It is similar in quality to Gemini Frontier-class models."— Apple Machine Learning Research, AFM 3 release page

What this changes

The disclosure resolves a year of speculation about how Apple would scale Apple Intelligence past on-device limits. The answer is hybrid by design: Apple keeps the privacy-routing layer, the user-facing Siri experience, and the on-device tier; Google provides the frontier model underneath; NVIDIA provides the hardware attestation that lets Apple ship its privacy guarantees across a third-party cloud. The architectural concession is real — Apple is no longer the sole inference vendor for its premium AI feature set — but the trade is that Apple instantly gets Gemini-Frontier-class quality without operating the supply chain itself.

Microsoft's open-source SkillOpt treats an agent's skill markdown as a trainable object and lifts Codex accuracy by 24.8 points

VentureBeat · 2026-06-12 · Source ↗

SkillOpt scores +23.5 on GPT-5.5 in direct chat, +24.8 inside Codex's agentic loop and +19.1 in Claude Code, all without modifying model weights. A separate optimizer model converts scored rollouts into bounded add/delete/replace edits on the skill document; only edits that strictly improve a held-out validation are accepted.

Microsoft Research's SkillOpt landed in VentureBeat's Friday-morning slot as a 1,500-word writeup framing the open-source project as the first deep-learning-style optimizer aimed at the skill layer of an agent rather than the model layer. The pitch is that agent quality today depends as much on the prompt-engineering quality of the skill descriptions handed to a frontier model as on the model's underlying capability — and that hand-tuning those skill documents is exactly the kind of repetitive search problem deep-learning optimizers are good at automating.

SkillOpt's central abstraction is the skill markdown file. The optimizer treats every line in a SKILL.md document as a trainable parameter. A separate small model — the optimizer — scores rollouts of the agent against held-out tasks, generates bounded add/delete/replace edits to the skill document, and accepts an edit only when it strictly improves validation accuracy. The training loop is gradient-free by construction; the model weights of the agent never change. Microsoft Research published the project to GitHub in May 2026 under a permissive license, and the repo has accumulated roughly 3,020 stars by mid-June.

"On GPT-5.5, SkillOpt lifts average accuracy by +23.5 points in direct chat, inside OpenAI's Codex agentic loop it jumps to +24.8 points, and even in Claude Code, it delivers +19.1 points."— Ben Dickson, VentureBeat coverage of SkillOpt, June 12

Why this matters operationally

The deltas SkillOpt reports are not marginal. A 19-point lift inside Claude Code is the same order of magnitude as the gap between a generic frontier model and a domain-fine-tuned model — and SkillOpt achieves it through skill-document optimization alone. For Microsoft, the strategic logic is to commoditize the agent layer the way SaaS commoditized infrastructure: if the skill document is the unit of agent capability and the optimizer is open, every Microsoft customer can compound the value of Foundry agents without buying frontier-tier inference time.

The technical innovation is that the optimizer is itself a model, and Microsoft trains it on a corpus of optimization trajectories collected from earlier SkillOpt runs across multiple agents. That gives the optimizer some transfer learning across skill types — a SKILL.md for a code-review agent and one for a customer-support agent share enough structure that the optimizer learns generalizable edit policies. The paper measures this as cross-domain optimizer transfer with an average +14.2 lift even on skill domains the optimizer had not seen during its own training.

Limits

SkillOpt is bounded by the host model. A skill document cannot teach a model knowledge it does not have, so the lifts top out when the agent task crosses the host model's capability frontier. The optimizer is also expensive at training time — Microsoft does not publish the per-run cost, but VentureBeat's writeup quotes a Microsoft Research source describing it as "comparable to a small fine-tune budget" per skill per optimization. The right framing is that SkillOpt is a search-time training cost that pays back across every inference of the resulting skill thereafter.

Claude Code ships 2.1.174 and 2.1.175 on Friday with a managed-model allow-list enforcement and a dozen long-tail fixes

Claude Code changelog · 2026-06-12 · Source ↗

Version 2.1.175 introduces enforceAvailableModels, the managed setting that hard-pins the Default model to the allow-list so user or project settings cannot widen it. Version 2.1.174 fixes the /model picker, the Fable 5 credits banner for enterprise accounts, Bedrock GovCloud inference profiles, background-session env inheritance, and adds VSCode usage-attribution breakdowns for cache misses, long context and subagents.

Anthropic shipped two Claude Code releases on Friday. Version 2.1.174 went out as a maintenance release covering twelve named bug fixes and one small feature addition, then 2.1.175 followed the same day with a single but consequential managed-settings change. The releases are the company's first material Claude Code update since the Fable 5 launch on June 9 and the Mythos 5 partner rollout earlier in the week.

The 2.1.175 change is the load-bearing one for enterprise customers. The new enforceAvailableModels managed setting changes the semantics of the existing availableModels allow-list. Until Friday, a corporate admin could publish an availableModels list, but a Default-model resolution could still resolve to a model outside the list if no available alternative matched. With enforceAvailableModels enabled, Default falls back to the first allowed model when the natural resolution would land outside the list, and user or project settings are blocked from widening the managed allow-list. The change closes the last route through which a developer could effectively bypass an admin-enforced model policy from inside an organization-managed Claude Code installation.

"Added enforceAvailableModels managed setting — when enabled, the availableModels allowlist also constrains the Default model, and user or project settings can no longer widen a managed availableModels list."— Claude Code 2.1.175 changelog entry

2.1.174's longer list

The earlier Friday release covers a thoroughly grab-bag set of fixes that surfaced during the post-Fable-5 rollout. The /model picker had been hiding the model family that Default resolves to, which surfaced for any admin testing an availableModels configuration. The same picker had a hardcoded Sonnet version label that persisted even when ANTHROPIC_DEFAULT_SONNET_MODEL was pinned to a different Sonnet checkpoint. The Fable 5 credits banner had been showing for enterprise accounts on usage-based billing where credits do not apply. Bedrock GovCloud regions had been deriving the wrong inference-profile prefix, and background sessions had been inheriting another session's ANTHROPIC_* provider environment from the shell that started the background daemon.

The single feature in 2.1.174 is the VSCode usage-attribution breakdown. The Account & usage dialog inside the VSCode extension now itemizes cache misses, long context (specifically Fable 5's [1m] tier) and sub-agent usage so a developer can see where their inference budget went without going to the Claude dashboard. That data was previously only visible in the web admin console.

Sub-agents one notch deeper

Version 2.1.172, shipped Wednesday, deserves a sentence too: Claude Code sub-agents can now spawn their own sub-agents up to five levels deep. The change makes Claude Code's agentic decomposition pattern materially more flexible — a top-level planning agent can spawn researchers, each of which can spawn verifiers, each of which can spawn one more level of search. Five levels is a higher ceiling than any competing agent framework, and combined with the workflow-orchestration primitives Anthropic shipped in May, it positions Claude Code as the most expressive of the open-bytecode agent runtimes.

Linx Security announced on Friday morning the first production-grade identity-governance integration on top of Anthropic's Claude Compliance API. The integration brings Claude inside the Linx platform as a fully governed application, with the same visibility, audit trail and access controls that enterprise security teams already apply to the rest of their application stack.

Linx is a New York-based 2023-founded startup positioning itself as the identity-security layer for hybrid workforces that mix human, non-human and AI identities. The company has backing from Insight Partners, Index Ventures and Cyberstarts. The Claude integration is the first commercial product to ship on Anthropic's Compliance API, which itself only went GA earlier this quarter.

"You can see who has access to Salesforce. You should be able to see the same for Claude."— Israel Duanis, Linx Security CEO and co-founder

What it actually does

The integration pulls four streams of Claude state into Linx's identity graph. First, it enumerates every Claude account inside an organization, the group memberships and the administrative roles attached. Second, it surfaces every API key under every project inside the enterprise's Claude deployment, with last-used and last-rotated timestamps. Third, it pulls project-level resource maps so a security team can see which folders, knowledge sources and MCP servers each project has access to. Fourth, the audit-trail layer captures every Claude interaction at the session level — who asked Claude what, when, against which API key, with what resources mounted.

The integration's payoff is that all four streams flow into Linx's existing certification and review workflows. An organization that already reviews Salesforce access quarterly under SOX controls can now review Claude access on the same quarterly cycle, with the same review queue, the same Slack notifications, and the same break-glass procedures. The compliance pitch is "no new control plane for Claude."

"Combining Linx's identity governance with Anthropic's AI to make faster, smarter certification decisions means moving beyond compliance checkboxes toward genuinely-informed access reviews."— Niv Goldenberg, Linx Security CPO and co-founder

Why this happened now

The integration responds to a problem enterprise customers have been escalating since Anthropic crossed $44 billion of annualized revenue in May. Claude went from a Tier-3 IT shadow app to a Tier-1 production application across most Fortune 500 deployments inside roughly nine months. The standard identity-governance vendors — SailPoint, Saviynt, Microsoft Entra ID Governance — had no first-class integration. Linx, having shipped the AI-identity governance vision a year ago, was the natural partner for Anthropic to certify against the Compliance API first.

Expect the larger identity vendors to follow inside the next two quarters. SailPoint has signalled a Claude integration on its 2026 roadmap; Microsoft already governs Claude inside Entra at the basic level through Microsoft 365 admin telemetry, but the deeper API-key-level visibility Linx ships today is still a gap inside the Microsoft stack.

Anthropic disputes Pliny-the-Liberator's Fable 5 jailbreak that surfaced a 120,000-character system prompt within 48 hours of launch

TechTimes · 2026-06-12 · Source ↗

A multi-agent decomposition attack combining Unicode tricks, long-context reference tracking, document framing and fiction framing claims to defeat the three safety classifiers protecting Fable 5 and leak the model's full system prompt. Anthropic denies the substance of the claim without contesting individual technique details, citing 1,000-plus hours of pre-launch bug-bounty testing against more than 30 known techniques.

The Friday news cycle around Claude Fable 5 turned on a disputed jailbreak. Researcher "Pliny the Liberator," a known red-team handle who has surfaced exploits against most frontier models over the past two years, published a thread claiming to defeat the safety classifiers around Fable 5 using a multi-agent decomposition attack. The technique chains four primitives — Unicode trick characters, long-context reference tracking through Fable 5's million-token window, document-style framing of the harmful payload, and fiction framing of the request as a hypothetical narrative — and then routes the decomposed pieces through several agent personas before reassembling the final output. Pliny published what appears to be Fable 5's full 120,000-character system prompt as proof.

Anthropic publicly disputed the claim within hours but stopped short of contesting each individual technique used. The company's posture is that the attack as constructed does not produce a usable jailbreak of the model's harm policy in the substantive sense — the classifiers around cybersecurity, biology and chemistry route those queries to Opus 4.8 rather than answering them — and that the system-prompt leak by itself is not a safety incident on the company's existing classification, because Fable 5's system prompt does not contain information that materially changes the model's policy enforcement.

"Anthropic ran over 1,000 hours of external bug-bounty testing pre-launch. More than 30 known jailbreak techniques were thrown at the model. The classifier architecture caught what it was designed to catch."— Anthropic statement on Claude Fable 5 jailbreak claim, June 12

What the classifier architecture looks like

Fable 5's safety stack ships three classifiers in front of the main model: one for cybersecurity, one for biology/chemistry/CBRN, and one for distillation/IP. When a query trips any of the classifiers, the system reroutes to Opus 4.8 — a less capable model with tighter content restrictions — rather than refusing outright. Anthropic published the design at launch as "graceful degradation" rather than refusal, on the theory that a user with a legitimate adjacent question gets some help rather than a blanket no. The architecture fires on under 5 percent of sessions in production. Anthropic claims zero universal jailbreaks emerged during pre-launch bug bounty.

What Pliny's attack actually achieves

The fairest read of Pliny's published evidence: the multi-agent decomposition does not bypass the classifier system in the sense Anthropic was designed to defend. What it appears to bypass is the system-prompt secrecy layer, which is a softer guarantee that Anthropic had not heavily invested in. The system prompt itself is the kind of artifact most labs prefer not to publish but does not directly enable a harmful capability — it surfaces instruction patterns and tonal guidance, not weights or training data.

Where the dispute matters is the broader signal it sends about post-launch hardening. Bruce Schneier's open critique from the June 9 launch — that Anthropic's Project Glasswing has surfaced 23,000 vulnerabilities across 1,000-plus projects with almost none patched — sits in the background. Friday's incident is the first material public test of whether Anthropic's response loop on Fable 5 will look like the slow Glasswing one or like a same-day hot-patch culture. The next 72 hours of Anthropic's public posture will be the data point.

Jefferies' Blayne Curtis tells buy-side AMD's Helios MI400 rack likely slips to 2027 on UA Link supply

Jefferies client note · 2026-06-12 · Source ↗

Curtis tells clients UA Link will not be ready for production in 2026, implying AMD's full Helios rack with 72 MI400 GPUs slips out of the originally targeted second half this year. The MI400 chip itself remains on track; the scale-up rack assembly waits on Marvell and Astera Labs UALink switches. Tomahawk 6 Ethernet is the fallback.

Jefferies equity analyst Blayne Curtis sent buy-side clients a Friday morning note that AMD's Helios MI400 rack-scale platform will likely not ship for production in 2026. The supply constraint is on UA Link (Ultra Accelerator Link) switching technology, the interconnect AMD has positioned as its NVLink competitor for scale-up GPU pods. Curtis's checks indicate that Marvell and Astera Labs, the two named UALink switch ASIC suppliers, will not have production-ready silicon before the end of the calendar year. AMD's MI400 chip itself remains on its existing schedule; the bottleneck is the rack-level assembly that requires the switch fabric to be available in volume.

"Our checks suggest that UA Link will likely not be ready for production in 2026, implying this release [of the Helios rack] may end up being pushed out into 2027 as we've previously noted."— Blayne Curtis, Jefferies, June 12 client note

The fallback plan AMD already disclosed

AMD's CES 2026 disclosure included a Plan B for exactly this scenario. The MI400 scale-up network will use Broadcom's Tomahawk 6 Ethernet switches as a temporary alternative while UALink ramps. The trade-off is bandwidth and latency. UALink targets 1.8TB/s of bidirectional bandwidth per GPU at scale-up tier; Tomahawk 6 Ethernet on a 1.6Tbps SerDes lane configuration tops out closer to 800GB/s of equivalent scale-up bandwidth, with worse tail latency on collective operations. AMD's marketing materials and benchmark numbers for the Helios MI400 platform were built around the UALink performance envelope; using the Tomahawk fallback materially changes the comparative case against NVIDIA's Vera Rubin NVL144.

What this means for the AI capex cycle

The two-quarter Helios slip is the second major AI hardware schedule revision in the past month, following Oracle's Stargate Abilene Phase 2 cancellation. The pattern emerging across the buy-side is that the 2026 AI build-out is now supply-constrained at the interconnect and packaging layers rather than the GPU layer. NVIDIA's existing NVLink ecosystem, with mature switches from in-house designs, becomes structurally more valuable in a year where third-party scale-up fabric is the gating supply factor. The Jefferies note explicitly calls this out: "the AI rack TAM in 2026 looks materially more NVIDIA-skewed than the 12-month rolling view suggested entering Computex."

For AMD, the trade is bounded but real. The MI400 chip itself is still on track, and individual MI400 GPUs can ship through 2026 in OEM 4-socket or 8-socket boards for the existing data-center customer base. The full Helios rack-scale platform — the product AMD has positioned as its head-to-head competitor against NVIDIA's full GB200 NVL72 / Vera Rubin NVL144 stack — is what slips. AMD's commercial guidance for 2026 has not yet been revised; expect that on the next earnings call.

NVIDIA blog · 2026-06-12 (rollout phase) · Source ↗

The system attests trust in unmodified NVIDIA processors, encrypts data transmission, and supports remote attestation before any sensitive data leaves the device.

The architectural underpinning of Apple's third-generation Cloud Pro tier is the NVIDIA Confidential Computing stack running on Blackwell B200 GPUs inside Google Cloud. The setup means a request from an iPhone running iOS 27 lands inside a hardware-attested trusted execution environment on a third-party data-center GPU, and the user's privacy guarantee chains all the way back to Apple's signing infrastructure without Apple, Google or NVIDIA being able to inspect the data in transit or at rest.

NVIDIA's confidential computing technology achieves this through four mechanisms: it establishes trust in genuine, unmodified NVIDIA processors via a manufacturer-issued attestation; it encrypts data transmission between the host system and the GPU enclave; it enables remote attestation so the Private Cloud Compute system can verify the GPU's security state before submitting sensitive data; and it maintains the GPU's full performance characteristics for privacy-sensitive workloads.

"No one, not even the system's builders, can look at their data, chats or conversations."— NVIDIA, Confidential Computing for Apple PCC announcement

Why this is novel architecture

Apple's existing Private Cloud Compute architecture ran inside Apple-owned data centers on Apple Silicon servers. The trust model was simple: Apple controls both the hardware and the software, attests the configuration at boot, and publishes the cryptographic root-of-trust externally so security researchers can audit. Extending PCC to a third-party cloud running third-party hardware required a new attestation pattern in which Apple's PCC trust chain accepts the NVIDIA-issued attestation as a peer root and adds Google Cloud's infrastructure-attestation as a third anchor. The combined attestation is what gives Apple the technical basis to claim the PCC privacy guarantees survive the move to Google Cloud.

The rollout is gradual through summer 2026. Full operational capacity is expected by end of summer, with the AFM 3 Cloud Pro tier ramping over July and August as initial Blackwell B200 capacity comes online. The capacity is dedicated to Apple by Google under contract terms not disclosed in the announcement.

Strategic read

NVIDIA wins twice. Apple's choice of Blackwell-based attested execution sets a public reference architecture that other privacy-sensitive consumer-AI deployments — banks, healthcare, governments — can adopt without rebuilding the attestation stack from scratch. Each adoption pulls more Blackwell-tier purchasing through both NVIDIA and the hosting cloud. Google wins the hosting revenue and the strategic positioning as the cloud that Apple chose for its most privacy-sensitive AI workload. Apple gets Gemini-Frontier-class quality without operating the supply chain itself. The three-way arrangement is the cleanest signal yet that the privacy-attested-cloud-AI architecture has become a deployable reality at production scale.
