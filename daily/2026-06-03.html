Microsoft ships seven MAI models and Project Polaris, pulls Copilot off OpenAI's stack at Build 2026

Satya Nadella opened Microsoft Build 2026 in San Francisco on Tuesday with a model family Microsoft trained from scratch on its own Maia accelerators. MAI-Thinking-1, a 35-billion-parameter reasoning model, matches Claude Opus 4.6 on the SWE Bench Pro coding benchmark; MAI-Image-2.5 entered Arena.ai's image leaderboard at number three; and Project Polaris will replace GPT-4 Turbo as the default engine across every paid GitHub Copilot tier starting in August. The Copilot transition ends Microsoft's three-year dependence on OpenAI as the substrate for its most-shipped developer product.

Thurrott / Tech Times / Microsoft Foundry · 2026-06-02 · Source ↗ ·

Satya Nadella opened the Microsoft Build 2026 keynote at Fort Mason Center in San Francisco at 9:30 a.m. Pacific on Tuesday and spent the first thirty minutes of the program walking the audience through a frontier-model family Microsoft trained itself. The MAI line, formally introduced through the Microsoft Foundry blog the same morning, contains seven models that together cover reasoning, code, image generation, image editing, speech transcription, speech synthesis, and the Copilot default engine. Microsoft describes each model as "trained from the ground up on clean data, without distillation from third-party models," a phrase the launch material returned to repeatedly. The strategic message landed before the technical detail: Microsoft is no longer content to be the best enterprise distributor of someone else's AI.

The headline release is MAI-Thinking-1, a 35-billion-parameter reasoning model that Microsoft says matches Anthropic's Claude Opus 4.6, released in February, on the SWE Bench Pro coding benchmark. The model is in private preview through Microsoft Foundry to a select group of early partners. Microsoft characterizes Thinking-1 as a "medium-sized" model designed for high efficiency at a low per-token cost, which is the wedge it wants to push against frontier-priced offerings from OpenAI, Anthropic and Google. The company has not yet disclosed token pricing or context length, which will follow the public-preview window.

Polaris and the end of the GPT-4 era inside Copilot

Project Polaris is the second strategic release and the one with the most immediate revenue consequence. Polaris is a mixture-of-experts coding model with language-specific sub-modules that Microsoft says outperforms GPT-4 Turbo on HumanEval and MBPP, the two benchmark suites that have defined coding-model comparison since 2023. Microsoft has not published independent verification for those numbers, and the Tech Times writeup of the keynote explicitly notes the benchmark figures are Microsoft's own. Polaris runs on Microsoft's custom Maia AI accelerators inside Azure. Beginning in August 2026, Polaris will replace GPT-4 Turbo as the default engine for every paid GitHub Copilot tier. An optional three-month fallback to GPT-4 will be available before the August deadline.

GitHub CEO Thomas Dohmke described the Polaris transition as "the biggest change to Copilot since launch," a phrase that landed against an audience that remembers Copilot's October 2021 debut. The default-model swap is the single largest concrete commitment of the morning, and it ends the substrate dependency that has defined Microsoft's developer-tooling product since the original Codex partnership. Copilot users will be able to opt back to OpenAI or Anthropic models via the model-picker, but the default is the substrate that drives ninety-plus percent of usage in practice.

The image, voice, transcription, and code family

MAI-Image-2.5 entered Arena.ai's leaderboard at number three for image-generation model families, beating Google's Nano Banana Pro on ELO. The model ships in two variants, a high-quality version and a faster MAI-Image-2.5e variant, and accepts image uploads, which opens the model to image-editing as well as generation. Pricing in the Foundry Model Catalog starts at $5 per million tokens for text input, $8 per million tokens for image input, and $47 per million tokens for image output. MAI-Image-2.5 introduces identity and character consistency, preserving recognizable faces, hair, clothing, and full-body identity across stylization, pose, and layout changes, which is the feature that drove the Arena rating.

MAI-Voice-2 is the second-generation speech synthesis model, paired with a faster Flash variant. MAI-Transcribe-1.5 is the new speech-recognition model that Microsoft says is "five times faster than competing models" and offers "state-of-the-art accuracy" across 43 languages. MAI-Code-1 is the production coding model that ships directly into GitHub Copilot and Visual Studio Code, separate from Polaris (which is the orchestration default). The launch lineup also touches the third-party catalogue: Microsoft announced that Fireworks AI is now generally available on Microsoft Foundry with enterprise SLAs and SOC2 readiness, giving developers a single Azure endpoint into more than twenty open frontier models including DeepSeek, Kimi, GLM, gpt-oss, and MiniMax.

"MAI-Thinking-1 was trained from the ground up on clean data, without distillation from third-party models."— Microsoft Foundry launch material, June 2, 2026

The strategic frame

The MAI launch is the moment Microsoft completes a three-year repositioning. The company entered 2023 as the distribution channel for OpenAI's frontier models; it exits Build 2026 as a model maker that owns the runtime (Foundry), the silicon (Maia accelerators in Azure datacenters and the RTX Spark superchip on the client), the agentic SDK (Microsoft Agent Framework), and the productivity surface (Copilot across Office and GitHub). The OpenAI partnership remains, and OpenAI's Codex models continue to be available through Copilot and Foundry, but the default has shifted. Microsoft no longer depends on OpenAI's frontier-model release cadence to keep Copilot competitive.

The Polaris transition specifically tracks a pattern that began in March when GitHub Copilot CLI reached general availability with multi-model support and accelerated through Cursor's, Zed's, and Anthropic's Claude Code product entries during Q1. The agentic-coding category has compressed from a single OpenAI-dominated market into a multi-model contest in which Microsoft, Anthropic, and OpenAI each ship competitive offerings. By owning the default, Microsoft captures the margin tier that matters most in a metered-pricing environment, the tier that ships before the user changes any settings.

"The biggest change to Copilot since launch."— Thomas Dohmke, GitHub CEO, on the Polaris default transition

What the developer audience saw next

The MAI announcement opened a keynote that ran another two hours and covered Project Solara, the new agent-first device platform; Azure Agent Mesh, the federated agent-orchestration service; Aion 1.0, the new on-device Windows model line; WSL 3, with paravirtualized GPU and NPU passthrough; the standalone GitHub Copilot desktop app; Microsoft Execution Containers for agent isolation; Coreutils for Windows; the Arm Development Kit 2026; and the Surface RTX Spark Dev Box. The MAI family is the substrate. The rest of the keynote is what Microsoft built on top. By 12:30 p.m. Eastern when Nadella handed the stage to GitHub's session, Microsoft had announced more shippable AI surface area in a single morning than at any developer conference since the 2017 Azure Stack debut.

Trump signs frontier-model executive order, 30-day pre-release review, no mandatory licensing

CNBC / The Register / White House · 2026-06-02 · Source ↗ ·

"Promoting Advanced Artificial Intelligence Innovation and Security" creates a voluntary framework for federal pre-release testing of covered frontier models, shortens the review window from a May draft 90 days to 30 days, and explicitly bars any mandatory licensing or permitting. Anthropic and Cisco named as cooperating partners; CDT and the Cato Institute flagged the "trusted partners" mechanism as a political risk.

President Trump signed the executive order "Promoting Advanced Artificial Intelligence Innovation and Security" on the afternoon of June 2 at the White House. The order codifies a voluntary collaboration framework under which AI developers may share covered frontier models with federal agencies "up to 30 days before they plan to release such models to other trusted partners." The 30-day window is a deliberate reduction from the 90-day draft circulated through Axios in mid-May. Section 3(c) of the published order states that nothing in the document authorizes "mandatory governmental licensing, preclearance, or permitting" for AI model development, a clause inserted at the request of industry counsel after the draft drew pushback.

The benchmarking process for what counts as a covered frontier model is delegated to the NSA Director, in consultation with the Secretaries of War, Homeland Security, Treasury, the Attorney General, and the Secretary of Commerce. The benchmarking workstream is classified. The order also creates an AI cybersecurity clearinghouse through the Treasury Department, expands US Tech Force cybersecurity hiring, and directs federal grant programs toward AI vulnerability detection.

Industry posture and named partners

Anthropic is named directly in industry reporting as a participating partner, with the company's existing Project Glasswing federal-collaboration structure (covered separately in today's edition) cited as the pre-existing template. Cisco was the first major industry voice to praise the initiative publicly. The CNBC and NPR writeups of the signing identified Anthropic, OpenAI, and Google as the three frontier labs the voluntary framework most directly targets, although the order itself does not name individual companies.

"The government's involvement in decisions about which 'trusted partners' can access these advanced models gives the executive a great deal of discretion and could open the door to potential weaponization."— Juan Londoño, Cato Institute, on the EO's trusted-partner provision

The pushback

Civil-liberties and tech-policy groups raised three concerns. Samir Jain of the Center for Democracy and Technology said the order "should not become a mechanism for the Administration to punish companies for political or other arbitrary reasons." Neil Chilson, a former FTC attorney, said the discretionary gaps "could be used to pick winners and losers, or to give short-term national security concerns excessive weight." Juan Londoño at the Cato Institute focused on the trusted-partner mechanism. The order arrives six months after the December 2025 federalization order that established the AI Litigation Task Force, and three days after a federal magistrate stayed enforcement of Colorado's SB 24-205, the only major state AI-discrimination statute scheduled for 2026 enforcement.

"This order should not become a mechanism for the Administration to punish companies for political or other arbitrary reasons."— Samir Jain, Center for Democracy and Technology

The voluntary-versus-mandatory frame

The order is the second piece of the administration's federal-preemption strategy. The December 2025 executive order on national AI policy established the litigation infrastructure to challenge state AI laws; the June 2 order establishes the federal access channel for covered-frontier-model testing while explicitly disclaiming mandatory licensing. Industry voices read the disclaimer as the binding constraint and read the voluntary framework as the channel through which "covered frontier model" gradually becomes a regulated category. The classified benchmarking workstream means the public will not see the threshold for what counts as covered until the NSA's first report.

Microsoft technical fellow Stevie Bathiche walked the Build 2026 audience through Project Solara, a new chip-to-cloud platform for agent-first devices that ship a hardware form factor and an operating-system layer purpose-built to host AI agents rather than traditional applications. Solara runs on the Microsoft Device Ecosystem Platform, the company's enterprise fork of the Android Open Source Project, layered with a custom agent shell, Microsoft Intune device management, Entra ID identity, and Windows Hello for Business. The platform allows partners to ship MediaTek and Qualcomm Arm-based reference designs without rewriting the agent runtime.

Two concept devices anchored the demonstration. The first is a wearable smart-key badge with 5G connectivity, a touchscreen, a camera, and voice input, intended for frontline-worker scenarios where the badge becomes the user's primary computing interface. The second is a desk companion device, an Echo Show-style stationary unit with adaptive screens, microphones, cameras, a presence sensor, and USB-C ports that surfaces Microsoft 365 data including Outlook events and Excel content with voice input. Both designs are reference, intended for partner OEMs to manufacture rather than for Microsoft to ship at retail.

Just-in-time UI and the no-default-agent posture

The distinguishing technical feature is what Microsoft is calling "just-in-time UI," the platform's ability to render agent-driven interfaces dynamically across device sizes and modalities without requiring developers to design separate experiences for each form factor. The platform is also explicitly designed without a single dominant agent. Users select preferred agents manually rather than booting into a Microsoft-branded default. Microsoft has signaled that a future release will add an "agent dispatcher and agent task manager" layer that routes intents across multiple installed agents.

"Agents and AI accelerate the creation of even more specialized computers without incurring the full cost and tradeoffs that previously limited device diversity."— Stevie Bathiche, Microsoft Technical Fellow, on Project Solara at Build 2026

The pilot list

The external pilot will run with AccuWeather, Best Buy, CVS Health, Levi's, and Target as the named launch partners, with additional partners to follow in subsequent months. Microsoft has not announced consumer availability for either reference device, and the timeline pushes against Surface and Copilot+ PC categories that Microsoft would not want Solara to cannibalize. The framing in the launch material treats Solara as an enterprise channel: Intune and Entra ID management, security review for agent publication, and a developer-revenue model in which Microsoft takes a low platform fee through a curated Windows Agent Store.

Why Android, why now

The choice to build Solara on AOSP rather than on a Windows kernel surprised some attendees and surfaced a strategic logic Microsoft has been hinting at for two years. Android gives Solara immediate driver compatibility across the hardware ecosystem and lets Microsoft layer its enterprise identity and management stack on top without rebuilding the silicon-to-OS stack. The MDEP layer is the abstraction that makes the Android foundation invisible to the agent developer; the device feels like a Microsoft-branded computer running Microsoft-branded agent runtimes. Solara is therefore not a smartphone or tablet in any conventional sense; it is a category Microsoft is attempting to define from scratch around the proposition that the agent, not the app, is the new primitive.

Azure Agent Mesh federates agent execution across Windows servers, Windows 365 Cloud PCs and Azure Arc edges

aitoolsrecap / Windows News · 2026-06-02 · Source ↗ ·

Mesh is a control-plane that routes each agent task to the nearest available compute node based on latency and GPU availability, with the same API for developers regardless of target. Consumption-priced with a dedicated agent compute SKU. General availability is Q4 2026.

Azure Agent Mesh is the federation layer Microsoft built to solve the multi-site deployment problem that hits enterprise agents the moment they leave the prototype machine. An agent that works on a single workstation needs significant rework to run across 500 machines spread across three geographies. Mesh abstracts the deployment topology: developers target a single Mesh endpoint with the same APIs they use locally, and Mesh dispatches each agent task to the nearest available compute node, factoring latency, GPU availability, and the agent's declared data-residency constraints.

The control plane spans three Microsoft compute substrates. On-premises Windows servers running the new Windows Agent Framework runtime, Windows 365 Cloud PCs that provide managed multi-step agent compute under Microsoft governance, and Azure Arc-enabled edge devices for industrial and field-service scenarios where the agent runs on hardware Microsoft does not own. Each substrate exposes the same Mesh-compatible agent runtime, so the developer's deployment manifest declares capability and policy requirements; Mesh handles placement.

Pricing and packaging

Pricing is consumption-based with a dedicated agent compute SKU that prices per invocation rather than per virtual-machine hour. The SKU separates agent workloads from general Azure compute billing, a meaningful change that makes the cost of an agent-heavy workload visible at the line-item level for FinOps teams. General availability is scheduled for Q4 2026, with a preview channel opening to existing Azure customers in the weeks following Build.

"Mesh handles routing, dispatching each agent task to the nearest available node based on latency and GPU availability."— Microsoft Build 2026 Azure Agent Mesh briefing

The competitive frame

Mesh is Microsoft's direct response to LangGraph Cloud, Modal Labs' agent-deployment pattern, and the Anthropic Claude Code GitHub Actions pattern. The differentiator is policy integration: an enterprise that already uses Intune to manage Windows devices and Entra ID for identity can apply the same governance to agent execution across all three Mesh substrates without rebuilding policy infrastructure. The unified API across on-premises, cloud, and edge is the value proposition the launch material returned to repeatedly. The question Mesh leaves open is what happens to enterprises whose existing agent runtimes do not yet target the Windows Agent Framework manifest format; Microsoft signaled that an MCP-compatible shim will ship in preview during the fall.

What Mesh does not replace

Mesh does not replace Foundry Agent Service, the existing hosted-agent runtime for cloud-only workloads, and it does not replace Microsoft Agent Framework as the SDK. The trio is layered: Microsoft Agent Framework is the SDK developers write against; Foundry Agent Service is the hosted runtime for single-region cloud deployments; Mesh is the multi-site federation control plane. The same agent code targets all three; the deployment manifest changes. The architectural bet is that any enterprise serious about agentic AI will eventually need the Mesh substrate to scale beyond a single region.

WSL 3 lands at Build with paravirtualized NPU and GPU passthrough, ships through Windows Update

Tech Times / Neowin · 2026-06-02 · Source ↗ ·

The third generation of the Windows Subsystem for Linux replaces the Hyper-V virtualization boundary with paravirtualized hardware access, giving Linux PyTorch, JAX, and CUDA workloads near-native performance on Snapdragon X Elite Gen 2, Intel Meteor Lake / Lunar Lake, and incoming AMD Copilot+ silicon.

WSL 3 is the largest architectural change to the Windows Subsystem for Linux since WSL 2 introduced the Hyper-V kernel six years ago. The earlier model ran a Linux kernel inside a Hyper-V virtual machine, which gave Linux processes access to the GPU but with a virtualization overhead that made high-throughput inference and training workloads slower on WSL than on bare-metal Linux. WSL 3 replaces that boundary with a paravirtualized hardware-access layer in which the Linux kernel communicates directly with the Windows GPU and NPU drivers through a shared-memory interface.

The performance claim is "near-native" for Linux AI workloads, framed in the launch material against PyTorch, JAX, and CUDA-dependent libraries. The practical effect is that Windows-on-NPU developers no longer face the historical choice between booting Linux for the ML stack or working around Windows-side toolchain gaps. A developer writing a PyTorch model on a Snapdragon X Elite Gen 2 device can train and run inference inside WSL 3 with the NPU available at near-native throughput; the same model can be invoked from a Windows-native agent runtime over a Windows AI API call that bridges to the WSL kernel.

Hardware support and rollout

WSL 3 GPU and NPU passthrough is live at launch on Qualcomm Snapdragon X Elite 2 and on Intel Meteor Lake and Lunar Lake silicon. AMD Ryzen AI support is coming later in the year. The minimum hardware requirement is a 40-trillion-operations-per-second NPU, the same baseline that defines a Copilot+ PC. The distribution channel is Windows Update, consistent with how WSL 2 has historically shipped. Developers do not need to install WSL 3 separately; the upgrade flows through the same servicing pipeline as the rest of the Windows AI runtime.

"A Windows developer on a Qualcomm or Intel Copilot+ PC now has a path to GPU and NPU-accelerated Linux inference that requires no secondary OS and no hardware compromise."— Tech Times Build 2026 WSL 3 analysis

What this unblocks

The change unblocks the developer pattern that has been the friction point for two years. ML researchers and applied-ML engineers have largely stayed on Linux because WSL 2's GPU performance was good enough for development but not for production-grade training. The WSL 3 paravirtualized model closes the throughput gap. Combined with the Aion 1.0 in-box models, the Windows AI APIs surface, and the Foundry Local runtime that reached general availability in April, the WSL 3 launch positions Windows as a credible end-to-end ML development environment on consumer hardware, which is what the Build 2026 launch material asserts.

Windows Developer Blog · 2026-06-02 · Source ↗ ·

The Windows Agent Framework runtime ships in preview for Windows Insiders this month, integrating agents with the Windows shell, scheduler, and security model as system services. The Windows Agent Framework is the system-level agent runtime that gives third-party agents the same status as Windows applications. The framework launched in preview for Windows Insiders the day of Build with text-based agent support; multi-modal capability follows in subsequent monthly releases. The architectural change is that an agent published against the framework can register with the Windows shell, the task scheduler, and the security model as a managed service rather than as a sandboxed app or a browser-hosted runtime.
