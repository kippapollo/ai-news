Google Cloud makes AlphaEvolve generally available with sixteen enterprise wins named at launch

The evolutionary coding agent that lifted Google Spanner write amplification by 20 percent and dropped Willow quantum-simulation error rates tenfold on internal workloads went public on Friday inside the Gemini Enterprise Agent Platform, with results from BASF, JetBrains, Klarna, PacBio, Schrödinger, Oak Ridge National Lab, and eleven other named customers folded into the announcement page.

Google Cloud Blog · 2026-07-10 · Source ↗

Read in full

Google Cloud moved AlphaEvolve out of the private preview it had run since December 2025 and into general availability on Friday, publishing a customer roster of sixteen named enterprises and eight Google-internal use cases that put concrete numbers behind an agent Anant Nawalgaria and Laurynas Tamulevičius described as a discovery engine rather than a coding assistant. The system lives on the Gemini Enterprise Agent Platform. A customer provides a seed program, a deterministic client-side evaluator, and a scoring function. AlphaEvolve's agentic harness then searches the surrounding algorithmic space using Gemini as the mutation engine, testing each candidate against the evaluator and evolving toward higher-scoring variants over hours or days of compute.

"AI is moving beyond acting as a productivity assistant that accelerates how we work to a discovery engine that expands what we can achieve. By autonomously navigating complex computational search spaces, tools like AlphaEvolve are helping researchers and engineers uncover breakthrough algorithms."— Pushmeet Kohli, Chief Scientist, Google Cloud and VP Science, Google DeepMind

How the loop works

The four-step deployment pattern that Google documented follows Define, Measure, Optimize, Apply. A customer picks a segment of an existing algorithm to designate as evolvable, writes a scoring function that ranks candidates by correctness, latency, memory or any other measurable property, and hands the pair to AlphaEvolve. The agent generates candidate programs through Gemini-driven mutation, compiles and executes each candidate against the client-side evaluator, and keeps the top scorers as parents for the next generation. Runs typically last from a few hours to several weeks, and the output is human-readable source code an engineer reviews before deploying to production.

The client-side evaluator is a hard requirement. AlphaEvolve does not judge candidates against a natural-language rubric or an LLM-as-judge score. Every candidate compiles, runs against the customer's own test harness, and returns a number. That constraint keeps the search grounded in real behavior and is why the results published on Friday span domains as unlike each other as chip design, quantum error correction, molecular simulation and demand forecasting.

"Grok 4.5 outperforms other frontier models on real-world professional tasks. On the GDPval+ benchmark of expert-created workplace reasoning tasks across the economy: Grok 4.5 achieves a 29 percent mean pass rate."— Snorkel AI, GDPval+ testing writeup

The 29 percent lead

Grok 4.5 pulled 29 percent mean pass on the 2,000 sampled tasks. GPT-5.5 hit 22 percent. Claude Opus 4.8 hit 21 percent. The lead concentrated in domains that demand professional judgment rather than raw code or math. Legal work: 40 percent for Grok, 27 to 28 percent for the field. Education: 58 percent for Grok, 35 to 42 percent for the field. Healthcare: 35 percent for Grok, 23 to 25 percent for the field. QA analysis: 37 percent for Grok, 19 to 27 percent for the field. Those numbers matter because GDPval+ is not a synthetic reasoning suite. The hallucination doubling

The same test surface produced the story's darker number. Grok 4.5's hallucination rate on the GDPval+ tasks was 54 percent, up from 25 percent on Grok 4.3. The doubling is not a random-error phenomenon. Snorkel's writeup attributes it to increased model capacity: the model knows more, so when it is wrong it is more confidently wrong, and it fabricates supporting detail that reads plausible to a non-expert reviewer.

"The model knows more, but it is also more confident when it is wrong."— Snorkel AI, GDPval+ testing writeup on Grok 4.5

Artificial Analysis ranked Grok 4.5 fourth on its Intelligence Index at a score of 54. Claude Fable 5 held first. GPT-5.5 took second. Claude Opus 4.8 took third. "Grok 4.5 has landed at number 1 on multiple benchmarks, performing better than expected."— Elon Musk, CEO, SpaceXAI

Token efficiency and the CursorBench contamination flag

xAI's own token-efficiency disclosure put Grok 4.5 at 15,954 output tokens on the SWE-bench Pro tasks against 67,020 for Opus 4.8, a 4.2x efficiency gap that translates to roughly 17x lower cost per task at published API rates. The composite picture that emerged from Friday: Grok 4.5 leads on real professional work by a real margin, hallucinates more than any current frontier model on the same tasks, requires output validation for any accuracy-dependent workflow, and prices at $2 input and $6 output per million tokens against Opus 4.8's $15 and $75. b9947 through b9950: developer plumbing

b9947 added a --output option to the llama.cpp CLI, letting scripts and CI pipelines direct completion output to a file without redirecting the entire process stdout. b9948 changed the CUDA implementations of ggml_top_k and ggml_argsort to process data in smaller chunks, a memory-pressure reduction that prevents overflow on the 96 GB HBM3 configurations Grok 4.5, Fable 5 and GPT-5.6 Sol inference now routinely runs on. b9949 added a cluster-parallel decode path for the OpenCL flash-attention kernel targeting Adreno, extending the July 1 Adreno GEMM/GEMV work that had left the Q1_0 and q4_k paths without a matching decode-time flash attention. b9950 added a unit test to llama-batch and cleared a Windows build regression.

b9951: the new ET backend

b9951 landed the initial ggml-et backend implementation, a new backend target maintained under the ggml-et prefix that the pull request description tied to an unnamed accelerator family expected to enter llama.cpp's supported backend set later in Q3. The commit added the backend registration path, a stub kernel set covering ADD, MUL, MUL_MAT, MUL_MAT_ID, SOFTMAX, RMS_NORM, ROPE, and GLU, plus the buffer allocation and copy primitives required for the backend to hold a full model weight set on device. Georgi Gerganov merged the pull request at 16:15 UTC.

"ggml-et: Initial ET backend."— llama.cpp Release Notes, b9951

b9952: DeepSeek V4 KQ mask

b9952 converted all KQ masks to F16 when flash attention is used and removed the zero attention bias term for DeepSeek V4, tightening the code path for the DeepSeek V4 official release that mid-July timing places this coming week. The zero-attention-bias removal drops a redundant tensor from the KV cache and shaves a small amount of memory bandwidth per token, and the F16 mask conversion aligns the mask dtype with the FA kernel input requirement rather than forcing an FP32-to-FP16 downcast per attention layer. Both are quality-of-life optimizations sized for the two-week release window DeepSeek confirmed on June 30.

b9956 and b9957: CLI resilience and server tools

b9956 wrapped the base-URL parser in the llama.cpp CLI with an exception catch, preventing a crash when a user passes a malformed server address. Before the patch, an invalid URL like "http://localhost:" without a port would trigger an unhandled exception and drop the shell. The patch surfaces a clean error message and exits cleanly.

b9957 improved the server's tool integration and removed the older apply_diff path, replacing it with a new edit-tool abstraction that separates the tool interface from the specific edit backend. The change lets Ollama, LM Studio, and Jan.ai plug their preferred edit tools into the server without patching llama-server's core loop, and it cleared the way for the b9946 Hexagon tiling work from the July 9 tag to propagate cleanly into downstream server builds.

The eight-tag window on July 10 puts the July 1 through 10 total at 34 tags, roughly triple the average two-week cadence of Q1 2026. Why cross-shell integration

The Skill format Anthropic published in April 2026 defines a manifest and an execution boundary that lets one agent's tools show up inside another agent's harness without either party owning the runtime. Google shipping AlphaEvolve as a Skill means the discovery agent runs equally well under Claude Code, Antigravity, Cursor, or any other harness that has implemented the format. The choice reads as a competitive signal: Google is unwilling to gate AlphaEvolve behind Antigravity alone, since the enterprise buyer base runs a mix of coding agents and would balk at a shell requirement.

Claude Code / Antigravity workflows

Inside Claude Code, the pattern is: select a segment of a function marked with a comment sentinel; invoke the AlphaEvolve Skill; hand it a scoring function that the developer has already written and unit-tested; wait for candidate programs to return; and merge the best one. Inside Antigravity, the same pattern runs with Antigravity's built-in benchmark scaffolding. Both shells surface a progress log while the run is active, and both write the returned candidates as pull requests against the developer's own branch.

"AlphaEvolve returns human-readable, optimized code."— Anant Nawalgaria and Laurynas Tamulevičius, Google Cloud launch blog

The Skill approach also gives Google's platform team a distribution channel that does not depend on the Gemini Enterprise console.
